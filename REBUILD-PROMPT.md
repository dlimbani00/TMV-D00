# Teammate Voices — Complete Application Rebuild Prompt

## What You're Building

An enterprise employee survey platform called **Teammate Voices** — a three-tier web application for creating, distributing, and analyzing employee surveys within organizational programs. Think Qualtrics/Culture Amp but purpose-built for intern programs, onboarding feedback, and employee voice initiatives.

**Tech Stack:**
- **Frontend:** React 18 + TypeScript + Vite (no Next.js, no Tailwind — custom CSS)
- **Backend:** Spring Boot 3.2 + Java 17 + JPA/Hibernate
- **Database:** Oracle 23 Free (Docker) with Flyway migrations
- **Infrastructure:** Docker Compose (3 containers: oracle-db, backend, frontend)

## Pages JSON Structure

Surveys store questions inline in a `pages` CLOB column as JSON. This is the canonical question source for multi-page surveys:

```json
[
  {
    "pageId": "PG-WELCOME",
    "title": "Welcome",
    "label": "Welcome",
    "description": "Thank you for participating.",
    "showDescription": true,
    "sortOrder": 1,
    "questions": [
      {
        "questionText": "Which department were you placed in?",
        "questionType": "Single Select",
        "questionLabel": "Department",
        "sortOrder": 1,
        "isRequired": true,
        "options": [
          { "optionText": "Engineering", "optionValue": 1, "sortOrder": 1 },
          { "optionText": "Product", "optionValue": 2, "sortOrder": 2 },
          { "optionText": "Design", "optionValue": 3, "sortOrder": 3 }
        ]
      },
      {
        "questionText": "What would have improved your onboarding?",
        "questionType": "Text Area",
        "questionLabel": "Improve",
        "sortOrder": 2,
        "isRequired": false,
        "options": []
      }
    ]
  }
]
```

**Question Types:** Single Select, Multiple Select, Rating Scale, Grid Rating Scale, Radio Buttons, Checkboxes, Single-line Input, Text Area, Sliding Scale, Static Text

**Question ID Convention:** Frontend generates composite IDs like `PG-WELCOME-q0`, `PG-WELCOME-q1` from pageId + question index. The backend LogicRuleValidator accepts both numeric DB IDs and composite page-based IDs.

### Key Business Rules (Middleware Layer)

**Survey Lifecycle:**
- DRAFT → can edit, delete, publish
- ACTIVE → cannot edit or delete, can clone/unpublish/close
- CLOSED → cannot edit or delete, can clone/reopen

**Program Lock:**
- Cannot edit/delete a program if any linked survey has status = ACTIVE

**Logic Rule Validation (LogicRuleValidator):**
- Structural: valid rule types (visible_if, required_if, skip_to, pipe_text), valid operators (equals, not_equals, greater_than, less_than, between, contains, not_contains, is_empty, is_not_empty, any_of), valid action types (show, hide, require, skip_to, pipe)
- Referential: all questionId and targetQuestionId must exist in the survey (checks both DB question IDs and pages JSON composite IDs)
- Business rules: no circular skip_to chains (DFS cycle detection), no hiding required questions, no duplicate rule IDs, no self-skip, no duplicate conditions on same question, max 200 rules per survey, max 20 conditions per rule, survey must be DRAFT to edit rules
- Rule-action compatibility: visible_if → show/hide, required_if → require, skip_to → skip_to, pipe_text → pipe

**Publish Workflow (SurveyWorkflowService):**
1. Validate survey has title, pages, and questions
2. Check email readiness (templates assigned?)
3. State change: DRAFT → ACTIVE
4. Send admin notification email
5. Write audit log entry
6. Return PublishResult with survey + email readiness + notification status

### Key Page Specifications

**SurveyEditor (5 tabs):**
1. **Details** — name, summary, status toggle, program select, start/end dates, anonymous toggle
2. **Form Builder** — multi-page question editor, question types dropdown, add/remove/reorder
3. **Form Viewer** — read-only preview (same renderer as SurveyResponder but no submit)
4. **Configuration** — 2 sub-tabs: Flow View (visual question flow with rule config) + Rules List (all rules table)
5. **Settings** — email template assignments, dispatch validation checklist, send test email

**SurveyResponder (standalone, no nav):**
- Loads survey via token (`/respond/:token`) or direct ID (`/survey/:id/respond`)
- Renders pages with Next/Back navigation
- Progress bar at top
- Rating Scale: horizontal button row (1-5) with equal-sized tiles
- Submit → POST to backend → thank you screen
- If survey not ACTIVE: shows "not accepting responses" message

**ProgramDetail:**
- Top white card: program info (template type, name, status pill, description, dates)
- Metrics row: Total / Completed (green) / Sent (blue) / Pending (gray)
- Bottom gray card: participant grid with search + status filter dropdown
- Status pills: Completed=green, Sent=blue, Opened=blue, Pending=gray, Failed=red, Expired=orange, Not Dispatched=light gray

**EmailTemplateEditor (4 tabs):**
1. **Details** — name, description, category, subject, from name, status
2. **Template Builder** — WYSIWYG (contentEditable) with toolbar: Bold/Italic/Underline/Link/Heading/List/Align + Insert Merge Field dropdown + Card Block button
3. **Preview** — rendered HTML with merge fields replaced by sample data
4. **Settings** — assign to programs/surveys with trigger type and delay days

**Merge Fields:** `{{participant_name}}`, `{{email}}`, `{{survey_title}}`, `{{survey_link}}`, `{{program_name}}`, `{{cohort}}`, `{{company_name}}`, `{{sender_name}}`
